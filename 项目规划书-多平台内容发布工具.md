## 1. 项目概述

### 1.1 项目背景

内容创作者面临一个核心痛点：**一次创作，多处适配**。在不同平台（公众号、知乎、B站、小红书等）发布内容时，每个平台对格式、排版、字数、标签、封面尺寸等要求各不相同。创作者需要花费大量时间手动调整，严重影响创作效率。

### 1.2 项目目标

设计并实现一个 **"一次编辑，多平台自动适配，一键发布"** 的内容分发工具。

**核心功能：**
- 用户在统一编辑器中输入内容（Markdown + 元数据）
- 系统自动将内容适配为各目标平台的格式（HTML富文本、纯文本、Markdown等）
- 支持一键发布到多个平台（含模拟发布模式）
- 架构设计支持轻松扩展新平台

### 1.3 目标平台（第一版）

| 平台 | 优先级 | 发布形式 | 难度 |
|------|--------|----------|------|
| 微信公众号 | P0 | API 发布 | 中等 |
| 知乎 | P0 | 浏览器自动化 | 较高 |
| B站（专栏） | P1 | API + 浏览器自动化 | 中等 |
| 小红书 | P1 | 浏览器自动化（RPA） | 高 |

## 2. 多平台深度调研

### 2.1 微信公众号

**平台特征：**
- 日活用户超过 5 亿，内容创作者超 2000 万
- 支持图文消息（富文本）、视频消息、音频消息
- 单篇图文最多 20000 字
- 封面图尺寸：900×383（2.35:1，首条）、200×200（次条）

**API 能力现状（2025-2026）：**

| 接口 | 功能 | 状态 |
|------|------|------|
| `/cgi-bin/freepublish/submit` | 发布草稿 | ✅ 可用（不显示在主页） |
| `/cgi-bin/freepublish/batchget` | 获取已发布列表 | ✅ 可用 |
| `/cgi-bin/freepublish/delete` | 删除已发布 | ✅ 可用 |
| `/cgi-bin/draft/add` | 创建草稿 | ✅ 可用 |
| `/cgi-bin/message/mass/sendall` | 群发（推送给用户） | ⚠️ 认证账号 |
| `/cgi-bin/material/add_news` | 上传永久图文素材 | ✅ 可用 |

**关键限制：**
- 2025年7月起，个人主体账号、未认证企业号的发布API权限被回收
- `freepublish` 发布的内容不显示在公众号主页历史消息中
- 群发接口对认证服务号：每月4次；订阅号：每日1次
- 2026年4月起，API群发内容可能不在主页显示（社区仍在确认中）

**格式要求：**
- 内容为 HTML 富文本（白名单标签限定）
- 图片需先上传至微信服务器获取 `image_id`
- 封面图需要单独上传
- 不支持外部CSS，仅内联样式或特定 class

### 2.2 知乎

**平台特征：**
- 月活用户超过 1 亿
- 支持文章（长文，Markdown/HTML）、回答（绑定问题）、想法（短动态）
- 文章支持 Markdown 和富文本编辑

**API 能力现状（2025-2026）：**

知乎于 2026 年 5 月正式开放开发者平台（developer.zhihu.com），标志着其 API 策略的重大转变。

| 阶段 | 时间 | 开放程度 |
|------|------|----------|
| 2013-2025 | 长达12年 | 几乎无公开API |
| 2026年5月 | 正式开放 | AI搜索、热榜、直答等功能 |
| 未来 | 逐步开放 | 内容管理接口待确认 |

**当前可用的 API 端点（v4）：**

| 端点 | 功能 |
|------|------|
| `POST /api/v4/articles` | 创建/发布文章 |
| `PUT /api/v4/articles/{id}` | 更新文章 |
| `POST /api/v4/questions/{id}/answers` | 发布回答 |
| `POST /api/v4/comments` | 创建评论 |
| `GET /api/v4/search/v2` | 内容搜索 |

**认证方式：** OAuth 2.0，权限粒度包括 `write_answers`、`write_comments`、`admin_space` 等

**格式要求：**
- 文章支持 Markdown 原生格式
- 回答支持富文本
- 图片需使用知乎图床上传
- 代码块有特定的渲染规则

### 2.3 B站（哔哩哔哩）

**平台特征：**
- 日活用户超过 1 亿
- 核心为视频平台，但专栏（图文）、动态功能也很活跃
- 专栏文章支持富文本编辑
- 动态支持纯文字、图文、视频分享

**API 能力现状：**

**官方开放平台（member.bilibili.com/open）：**
- 2025年5月更新开发者服务协议
- 采用 HMAC-SHA256 签名认证（v2.0）
- 视频列表查询、详情查询接口可用
- **专栏/图文内容发布 API 不完全公开**

**视频投稿关键接口（Web端）：**

| 接口 | 功能 |
|------|------|
| `POST /x/vu/web/cover/up` | 封面上传（base64） |
| `POST /x/vupre/web/typepre` | 分类预测 |
| `GET /x/vupre/web/tag/recommend` | 标签推荐 |
| `POST /x/vu/web/add/v3` | 视频投稿 |

**动态发布接口：**

| 接口 | 功能 |
|------|------|
| `POST /x/dynamic/feed/draw/upload_bfs` | 图片上传 |
| `POST /api.vc.bilibili.com/dynamic_svr/v1/dynamic_svr/create` | 纯文字动态 |
| `POST /x/dynamic/feed/create/dyn` | 富媒体动态（图文/投票） |
| `POST /x/dynamic/feed/create/draft` | 定时发布草稿 |

**关键限制：**
- 标题最长 80 字符
- 标签最多 12 个
- 视频描述最多 2000 字符
- 所有状态变更必须携带 CSRF Token
- 2025-2026 法律风险加剧：知名开源项目 `bilibili-API-collect` 收到律师函并清空代码

### 2.4 小红书

**平台特征：**
- 月活用户超过 4 亿（2026年数据）
- 内容以图文笔记为主，视频笔记快速增长
- 图文笔记最多 1000 字 + 最多 18 张图片
- 封面图：3:4 比例（推荐 1080×1440）
- 标签最多 10 个

**API 能力现状：**

**官方开放平台（open.xiaohongshu.com）：**
- 主要面向认证品牌方、MCN机构、电商商家
- **核心结论：不向普通开发者开放内容发布 API**

| 可申请权限 | 说明 |
|-----------|------|
| `note.read` | 读取笔记数据 |
| `product.read` | 商品数据 |
| `ad.report` | 广告报表 |
| 笔记发布 | ❌ **不开放** |

**可行的替代方案：**

| 方案 | 技术 | 风险 | 推荐度 |
|------|------|------|--------|
| **浏览器自动化（RPA）** | Playwright / Appium + 云手机 | 中等 | ⭐⭐⭐⭐ |
| **Spider_XHS** | 逆向协议 | 高封号风险 | ⭐⭐⭐ |
| **Airtest + 云手机** | 图像识别 + 模拟触摸 | 低 | ⭐⭐⭐⭐⭐ |

**2025-2026 趋势：** 非授权爬虫和自动化的风险持续升高，平台进入"白名单+接入"时代。

## 3. 市场需求分析

### 3.1 用户画像

| 用户类型 | 规模估计 | 核心痛点 | 付费意愿 |
|----------|----------|----------|----------|
| 个人自媒体 | 3000万+ | 每天花1-2小时手动跨平台排版发布 | 中等（月付30-50元） |
| 知识付费创作者 | 500万+ | 长文需适配各平台排版、代码块、公式渲染 | 高（月付50-100元） |
| MCN / 内容团队 | 10万+ | 多账号、多平台、多人协作 | 高（月付500-2000元） |
| 企业品牌号 | 200万+ | 品牌调性统一、合规审核 | 高（月付1000-5000元） |

### 3.2 市场规模

- 中国内容营销市场规模 2025 年预计超过 5000 亿人民币
- 内容创作工具赛道市场规模约 200-500 亿
- 跨平台内容分发是创作者工具链中最刚需的环节之一
- 75% 以上的自媒体运营者同时在 3 个以上平台发布内容

### 3.3 核心痛点量化

| 痛点 | 影响 | 量化指标 |
|------|------|----------|
| 格式适配 | 手动调整排版 | 单篇耗时 15-40 分钟 |
| 封面图适配 | 每平台尺寸不同 | 额外 10-15 分钟 |
| 发布操作 | 逐平台登录发布 | 额外 10-20 分钟 |
| 发布后管理 | 数据回收分散 | 每平台 5-10 分钟查看 |
| **总计** | **单篇多平台完整发布** | **40-85 分钟额外耗时** |

### 3.4 差异化机会

现有工具主要面向海外平台（Twitter/X、Facebook、LinkedIn等），**国内四大平台（公众号、知乎、B站、小红书）的一站式发布工具严重稀缺**，原因在于：

1. 国内平台 API 开放程度低
2. 格式差异大（HTML vs Markdown vs 纯文本）
3. 平台反爬和反自动化机制强
4. 合规风险高

**这恰恰是本项目的切入点和创新空间。**

### 4.2 我们的定位

```
              API开放程度

     高           │          低
     ┌───────────┼───────────┐
     │  Postiz   │           │
海   │  BrightBean│  本项目   │
```

**本项目的独特定位：**
1. **100% 开源 + 可自部署**（不同于 TurboPush 闭源核心）
2. **国内平台优先**（不同于 Postiz 海外优先）
3. **AI 驱动的格式适配**（语义级改写，而非简单格式转换）
4. **混合发布策略**（API优先 + 浏览器自动化兜底）
5. **可扩展的适配器架构**（新平台只需3个文件即可接入）

## 5. 技术栈选型

### 5.1 总体技术架构

| 层 | 技术选型 | 备选方案 | 选型理由 |
|-----|----------|----------|----------|
| **前端** | React 18 + TypeScript | Vue 3 / Next.js | 生态最丰富，组件库多，适合富文本编辑器集成 |
| **后端** | Node.js + Express (TypeScript) | Python FastAPI / Go Gin | 前后端统一语言，降低认知负荷，适合3日快速开发 |
| **数据库** | SQLite (dev) / PostgreSQL (prod) | MySQL / MongoDB | SQLite 零配置适合快速开发，PG 适合生产 |
| **ORM** | Drizzle ORM | Prisma / TypeORM | 轻量、类型安全、性能好 |
| **前端构建** | Vite | Webpack / Turbopack | 极快的开发服务器 |
| **UI组件** | Tailwind CSS + Radix UI | Ant Design / shadcn/ui | 原子化CSS + 无样式组件，定制灵活 |

### 5.2 核心技术决策

#### 5.2.1 富文本编辑器

| 方案 | 优势 | 劣势 | 推荐 |
|------|------|------|------|
| **Tiptap** | 基于 ProseMirror，高度可定制，支持 Markdown 快捷输入 | 学习曲线中等 | ⭐⭐⭐⭐⭐ |
| Slate.js | 完全可定制 | 文档少，API 变化大 | ⭐⭐⭐ |
| Quill | 简单易用 | 定制性差 | ⭐⭐ |
| Monaco Editor | 代码编辑体验好 | 非所见即所得 | ⭐⭐ |

**推荐：Tiptap** — 支持 Markdown 快捷输入的同时提供所见即所得体验，输出可为 HTML 或 JSON，方便格式转换。

#### 5.2.2 浏览器自动化

| 方案 | 优势 | 劣势 | 推荐 |
|------|------|------|------|
| **Playwright** | 跨浏览器、API 现代、微软维护 | 包体积大 | ⭐⭐⭐⭐⭐ |
| Puppeteer | Google 维护，轻量 | 仅 Chromium | ⭐⭐⭐⭐ |
| Selenium | 多语言、成熟 | API 老旧、速度慢 | ⭐⭐⭐ |

**推荐：Playwright** — 支持多浏览器，有 stealth 插件生态，API 设计优秀。

#### 5.2.3 格式转换引擎

| 方案 | 优势 | 劣势 |
|------|------|------|
| **自建适配器模式** | 完全可控，轻量 | 需要自行设计 |
| Pandoc | 支持格式多 | 过于重量，不适用Web场景 |
| unified/remark/rehype | AST级操作，插件丰富 | 学习曲线 |

**推荐：unified/remark/rehype 生态 + 自建适配器层**

- 使用 `remark` 解析 Markdown → MDAST
- 使用 `rehype` 将 MDAST 转为 HAST → HTML
- 使用 `unified` 管道串联转换步骤
- 自定义插件处理平台特定格式（如公众号不支持的外部CSS → 内联样式）

#### 5.2.4 AI 增强（可选扩展）

| 功能 | 方案 |
|------|------|
| 标题改写 | 调用 LLM API（Claude/GPT）根据不同平台风格改写 |
| 摘要提取 | 自动生成不同长度的摘要 |
| 标签推荐 | 基于内容的标签生成 |
| 去 AI 味 | 对 AI 生成内容做人性化处理 |

### 6.2 核心设计模式

#### 6.2.1 适配器模式（Adapter Pattern）

这是整个架构的核心。每个平台实现统一的 `PlatformAdapter` 接口：

```typescript
interface PlatformAdapter {
  // 平台元信息
  readonly id: string;           // "wechat" | "zhihu" | "bilibili" | "xiaohongshu"
  readonly name: string;         // "微信公众号"
  readonly icon: string;         // icon URL or component

  // 格式转换
  convertContent(input: UnifiedContent): Promise<PlatformContent>;

  // 发布策略
  publish(content: PlatformContent, credential: Credential): Promise<PublishResult>;

  // 能力查询
  getCapabilities(): PlatformCapabilities;
  validateCredentials(credential: Credential): Promise<boolean>;
}

interface UnifiedContent {
  title: string;
  body: string;             // Markdown
  excerpt?: string;         // 摘要
  coverImage?: string;      // 封面图 URL / base64
  tags: string[];
  category?: string;
}

interface PlatformContent {
  title: string;
  body: string;             // 平台原生格式（HTML/MD/纯文本）
  metadata: Record<string, unknown>;
}
```

#### 6.2.2 发布策略模式（Strategy Pattern）

```typescript
interface PublishStrategy {
  execute(adapter: PlatformAdapter, content: PlatformContent, credential: Credential): Promise<PublishResult>;
}

class ApiPublishStrategy implements PublishStrategy { /* 通过官方API发布 */ }
class RpaPublishStrategy implements PublishStrategy { /* 通过浏览器自动化发布 */ }
class SimulatedPublishStrategy implements PublishStrategy { /* 模拟发布，用于演示 */ }
```

#### 6.2.3 工厂模式（Factory Pattern）

```typescript
class AdapterRegistry {
  private adapters: Map<string, PlatformAdapter> = new Map();

  register(adapter: PlatformAdapter): void {
    this.adapters.set(adapter.id, adapter);
  }

  get(id: string): PlatformAdapter | undefined {
    return this.adapters.get(id);
  }

  list(): PlatformAdapter[] {
    return Array.from(this.adapters.values());
  }
}
```

### 6.4 扩展新平台架构

**只需3个文件即可添加新平台：**

```
adapters/new-platform/
├── index.ts          # 实现 PlatformAdapter 接口
├── format.ts         # 格式转换逻辑（平台特定规则）
└── publish.ts        # 发布策略实现（API or RPA）
```

**扩展步骤：**
1. 创建 `adapters/<platform>/` 目录
2. 实现 `PlatformAdapter` 接口的 4 个方法
3. 在 `adapters/index.ts` 中注册
4. 前端自动发现并显示新平台选项

## 9. 风险与注意事项

### 9.1 技术风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|----------|
| 微信公众号 API 权限限制 | 无法通过 API 发布 | 高 | 提前准备 RPA 方案作为降级 |
| 知乎 API 内容发布端点未完全开放 | 仅能检索，不能发布 | 中等 | RPA 兜底，关注 developer.zhihu.com 更新 |
| 小红书反自动化检测 | 封号风险 | 高 | 仅做模拟发布，RPA 策略加入随机延迟 |
| Playwright stealth 被检测 | 自动化失败 | 中等 | 使用 playwright-stealth，模拟人类行为 |
| B站 CSRF Token 过期 | RPA 发布失败 | 低 | Cookie 持久化 + 自动刷新 |

### 9.2 合规风险

| 风险 | 缓解措施 |
|------|----------|
| 违反平台 ToS | 仅用于个人学习/演示用途，README 中添加免责声明 |
| B站第三方API法律风险 | 使用官方开放平台优先，RPA仅作技术验证 |
| 用户数据安全 | 所有凭证本地加密存储，不上传云端 |

### 9.3 开发过程注意事项

1. **PR 规范：** 每个 PR 只做一件事，标题和描述必须清晰完整
2. **分支策略：** `main` 分支始终可运行，功能在 feature 分支开发
3. **依赖声明：** README 中列出所有第三方库及其用途和许可证
4. **原创说明：** 如引用自己过去的代码片段，在 PR 描述中注明来源
5. **代码质量：**
   - 函数 < 50 行，文件 < 800 行
   - 不可变数据模式
   - 显式错误处理
   - 类型安全（TypeScript strict mode）
