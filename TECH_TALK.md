# MultiPublish 项目历程与技术路线演讲文档

## 0. 演讲定位

**建议时长**：10 到 15 分钟
**听众假设**：老师、同学、评委或不了解项目细节的人
**一句话主题**：MultiPublish 是一个面向内容创作者的多平台发布工具，核心目标是把“一篇内容改多版、登多站、手动发布”的重复劳动，收敛成“一次编辑，多端适配，统一发布”的流程。

这次演讲不要只讲“我们做了哪些页面”，重点讲清楚三件事：

1. 为什么这个问题值得做。
2. 我们是怎么一步步把原型做成可演示产品的。
3. 技术上最关键的原理和后续路线是什么。

如果你担心技术名词听起来太硬，可以记住一个讲法：

**前端像操作台，后端像调度中心，Chrome 扩展像真正去各个平台执行发布动作的执行员，AI 代理像统一接不同 AI 服务的翻译层。**

整场演讲可以围绕这句话展开。不要一上来堆 React、Express、Chrome Extension、Provider Adapter 这些词，先讲“它们分别解决什么问题”，再补技术名词。

## 1. 项目背景：为什么做 MultiPublish

内容创作者经常需要把同一篇文章发布到微信公众号、知乎、B 站、小红书、微博等多个平台。每个平台的编辑器、图片上传方式、标题规则、正文格式、登录状态和发布入口都不一样。

如果人工处理，一篇内容通常要经历这些步骤：

- 复制标题和正文。
- 按平台重新排版。
- 重新上传或粘贴图片。
- 检查预览效果。
- 分别打开平台并点击发布。
- 回来记录每个平台是否发布成功。

这个过程不是一次性工作，而是高频重复工作。项目 README 里也总结过，一个创作者为了适配多平台，经常会在一篇内容上额外花几十分钟。MultiPublish 的目标就是把这些重复流程产品化、自动化。

从提交记录看，项目的演进非常集中，主要分成四个阶段。

### 第一阶段：从想法到基础架构，2026-05-29

这一天是项目的搭建期。提交记录里可以看到大量基础能力从零开始建立：

- 初始化前端和后端项目结构。
- 建立内容管理能力，包括文章创建、编辑、列表、详情等基础页面。

- 后端提供内容接口和发布接口。

这个阶段最重要的不是页面好不好看，而是确定了项目的技术方向：

**Web 端负责内容生产和任务编排，Chrome 扩展负责进入真实平台页面完成发布。**

第二天的提交量最多，说明项目进入了密集攻坚期。这个阶段主要解决的是“真实发布”问题。

关键变化包括：

- 编辑器体验持续重构，包括 Tiptap 富文本编辑、图片上传、图片拖拽缩放。
- 平台预览能力完善，让同一篇内容可以看到不同平台的展示效果。
- 微信、知乎、B 站、小红书等平台的真实发布逻辑逐步落地。

- 加入 AI 写作助手，用来辅助标题、摘要、正文润色和内容生成。
- 前端做了多轮 UI 重构，让项目从“功能集合”变成“可展示产品”。

这一阶段的核心挑战是：每个平台都不是为自动化发布设计的。页面结构会变，编辑器有自己的内部状态，图片上传还有异步过程。所以我们不是简单地 `set value`，而是针对平台分别处理 DOM、事件、剪贴板、文件输入框和前端框架状态。

- 增加发布记录、状态管理和错误提示。
- 完善 QuickStart、演示模式、进阶指引等面向演示和交付的内容。

这个阶段说明项目已经不是“某个平台能不能填进去”的局部验证，而是在形成一个完整链路：

**写作 -> AI 辅助 -> 多平台适配 -> 平台预览 -> 扩展接管 -> 真实发布 -> 记录结果。**

### 第四阶段：整理公开与近期修复，2026-06-08 之后

6 月 8 日的提交是公开仓库的初始整理提交，说明项目已经进入对外展示和维护阶段。

所以项目路线不是“写了一个多平台发布页面”，而是从基础 CRUD、真实平台注入、AI 能力、安全代理、演示稳定性一路迭代过来的。

## 4. 总体架构原理

可以用这张文字架构图解释：

```text
用户
  |
  v
Web 前端控制台
  |-- 内容编辑
  |-- 平台预览
  |-- AI 写作辅助
  |-- 发布任务管理
  |
  v
Web 后端 API
  |-- 内容 CRUD
  |-- 发布记录
  |-- AI 供应商代理
  |-- 参数校验和安全处理
  |
  v
Chrome 扩展
  |-- background 接收任务
  |-- chrome.storage.local 传递平台任务
  |-- content script 注入真实平台页面
  |
  v
目标平台编辑器
  |-- 微信
  |-- 知乎
  |-- B 站
  |-- 小红书
  |-- 微博
```

这里最关键的是：MultiPublish 没有假装自己能绕开平台后台接口，而是让 Chrome 扩展在用户已登录的真实页面里完成操作。

这样设计有三个好处：

- 不需要破解平台私有接口。
- 可以复用用户自己的登录态。
- 更接近真实创作者的操作路径。

### 4.1 小白版解释：一篇文章到底怎么发出去

可以拿“一篇文章发布到知乎和小红书”举例：

1. 用户先在 MultiPublish 的 Web 页面里写文章。
2. 前端把标题、正文、图片、平台选择这些信息整理成一个发布任务。
3. 如果用户点了 AI 辅助，前端不会直接访问 AI 平台，而是把请求交给后端。
4. 后端根据用户选择的 AI 服务，把请求转换成对应平台能识别的格式。
5. AI 返回结果后，前端把生成的标题、摘要或正文展示给用户。
6. 用户点击发布后，系统把“要发到哪个平台、发什么内容、当前任务 ID 是多少”记录下来。
7. Chrome 扩展收到任务后，打开或定位到真实的平台页面。
8. content script 进入平台页面，把标题、正文和图片填进去。
9. 平台发布完成后，扩展把成功或失败结果回传，Web 页面显示发布状态。

这里可以强调一点：

**MultiPublish 不是自己造一个假的发布平台，而是把真实平台上的重复操作自动化。**

### 4.2 架构图该怎么讲

讲架构图时建议按“从上到下”讲，不要按技术名词讲：

- 第一层是用户看到的页面，也就是 Web 前端。
- 第二层是后端，负责保存数据、校验数据、转发 AI 请求。
- 第三层是 Chrome 扩展，负责进入真实网页执行发布。
- 第四层是目标平台，比如微信公众号、知乎、B 站、小红书、微博。

你可以这样说：

> 我们把系统拆成几层，不是为了复杂，而是因为每一层权限不同。Web 页面适合做编辑和管理，但不能直接操作别的网站；后端适合保存数据和保护密钥；Chrome 扩展才有能力在用户授权后进入真实平台页面执行自动填充和发布。

## 5. 三个核心技术原理

### 原理一：统一内容模型 + 平台适配器

不同平台要求不一样，但我们不能让业务逻辑到处写 `if platform === xxx`。所以项目先把文章抽象成统一内容模型：

- 标题。
- 正文。
- 图片。
- 标签或话题。
- 平台配置。
- 发布状态。

然后每个平台通过适配器处理自己的特殊规则：

- 字段怎么映射。
- 正文要不要转换格式。
- 图片怎么上传。
- 发布按钮在哪里。
- 成功或失败怎么判断。

这样新增平台时，不需要推翻整体流程，只要新增一个平台适配器和对应扩展注入策略。

### 原理二：Chrome 扩展真实发布

Web 页面本身没有权限直接操作其他网站的 DOM，所以真实发布必须交给浏览器扩展。

核心流程是：

1. Web 前端发起发布。
2. 后端保存任务和状态。
3. 前端或扩展把平台发布任务写入 `chrome.storage.local`。
4. Chrome 扩展 background 监听任务。
5. 扩展打开或定位到目标平台页面。
6. content script 在页面内等待编辑器加载。
7. 注入标题、正文、图片等内容。
8. 点击保存、草稿或发布按钮。
9. 把结果写回 storage，Web 端展示发布状态。

这个设计的难点不在“复制文本”，而在“让平台认为这是用户正常输入”。很多现代编辑器只改 DOM 不会触发内部状态更新，所以必须派发真实事件，或者使用剪贴板、文件输入框、框架内部结构等方式让平台接收内容。

### 原理三：AI 后端代理与供应商适配

AI 功能不能把 API Key 放在前端。项目采用后端代理：

```text
前端 AI 请求 -> 后端 /api/ai/chat -> Provider Adapter -> 具体 AI 供应商
```

这样做有几个原因：

- API Key 不暴露在浏览器。
- 可以统一超时时间、错误格式和流式输出。
- 可以兼容不同供应商的参数差异。
- 后续可以增加审计、限流、缓存和费用控制。

不同 AI 平台虽然都叫“聊天接口”，但细节差异很多，比如：

- 接口地址不同。
- 模型命名不同。
- 有的用 `max_tokens`，有的用 `max_completion_tokens`。
- 有的模型对 `temperature` 有固定限制。
- 流式输出格式也不完全一致。

所以 Provider Adapter 的价值就是把供应商差异收敛到后端，前端只面对统一的 AI 能力。

## 6. 关键难点与解决办法

### 难点一：平台编辑器差异很大

微信、知乎、B 站、小红书、微博使用的编辑器和前端技术都不同。有的是富文本编辑器，有的是 React 状态驱动，有的有 shadow DOM，有的图片上传流程很复杂。

解决办法：

- 每个平台单独写注入策略。
- 等待页面节点稳定后再操作。
- 通过事件、剪贴板、文件输入框等方式模拟真实输入。
- 用发布结果回写机制判断成功或失败。

### 难点二：Web 端和扩展端不是同一个运行环境

Web 应用运行在本地服务里，扩展运行在 Chrome 的扩展上下文里，目标平台页面又是第三个上下文。

解决办法：

- 用 background 作为扩展任务中枢。
- 用 content script 进入目标平台页面。
- 用 `chrome.storage.local` 作为任务和结果的共享通道。
- 用唯一任务 ID 区分不同平台、不同文章、不同发布状态。

### 难点三：AI 接口容易因为供应商差异失败

多供应商接入看起来相似，实际参数、模型、错误码和流式格式都有区别。

解决办法：

- 所有 AI 请求都走后端。
- 后端使用 Provider Adapter 屏蔽差异。
- 统一 2 分半超时时间。
- 对模型参数做兼容处理。
- 对流式响应做统一转换。

### 难点四：演示项目不能只“功能存在”，还要“链路稳定”

比赛或汇报时，功能卡在半路比功能少更影响效果。

解决办法：

- 增加 QuickStart 和演示模式。
- 明确真实发布依赖 Chrome 扩展。
- 保留模拟发布能力，便于没有登录平台时展示主流程。
- 增加发布状态、错误提示和取消发布。
- 对前后端构建和测试做验证。

## 8. 后续技术路线

后续路线可以分成短期、中期、长期三层讲。

### 短期：提升稳定性

- 增加平台页面变化检测。
- 增强发布失败原因识别。
- 增加登录态检测。
- 优化图片上传成功判断。
- 增加更完整的端到端测试。
- 把每个平台的选择器和策略配置化，减少页面变化带来的维护成本。

### 中期：提升产品能力

- 支持定时发布。
- 支持团队协作和多人内容审核。
- 支持发布前的智能平台改写。
- 支持热点标题、摘要、标签自动生成。
- 支持内容素材库和图片复用。
- 支持发布效果统计和复盘。

### 长期：形成平台化能力

- 把平台适配器做成插件体系。
- 接入更多内容平台。
- 引入发布任务队列和重试机制。
- 引入权限、审计和费用控制。
- 支持 SaaS 化部署。
- 从“多平台发布工具”升级为“内容分发自动化平台”。

## 9. 可直接照着讲的详细演讲稿

大家好，我今天介绍的项目是 MultiPublish，一个多平台内容发布工具。

我先用一句话介绍它：MultiPublish 想解决的是内容创作者在多个平台重复发布内容的问题。比如一个人写完一篇文章，可能要发到微信公众号、知乎、B 站、小红书和微博。内容本身是一篇，但是每个平台都要重新打开、复制、排版、上传图片、检查格式，再点击发布。这个过程技术含量不一定高，但很耗时间，而且很容易出错。

所以我们做 MultiPublish 的目标就是：让用户在一个地方完成内容编辑，然后由系统帮他适配多个平台，并尽可能自动完成发布流程。简单说，就是“一次编辑，多平台触达”。

这个项目的开发不是一次性完成的。从提交记录和说明文档看，我们大概经历了四个阶段。

第一个阶段是 2026 年 5 月 29 日，这一天主要做基础架构。我们先搭建前端和后端，做内容创建、内容编辑、内容列表、发布接口和基础页面。这个阶段还引入了一个很重要的思想，叫平台适配器。它的意思是：系统内部先用统一的数据结构表示一篇文章，然后不同平台各自写自己的转换和发布逻辑。这样后面新增平台时，不需要把整个系统重写一遍。

这个阶段还确定了一个关键架构：Web 前端负责编辑、预览和管理任务，后端负责数据和接口，Chrome 扩展负责进入真实平台页面完成发布。这个分工非常重要，因为普通 Web 页面没有权限直接操作其他网站，所以真实发布必须交给浏览器扩展。

第二个阶段是 2026 年 5 月 30 日，这一天的重点是从“功能能跑”变成“真实平台能发布”。真实发布比普通页面开发难很多，因为每个平台的编辑器都不一样。比如有的平台是富文本编辑器，有的平台由 React 控制页面状态，有的平台上传图片要走文件输入框，有的平台必须模拟剪贴板事件。

如果只是简单地改网页上的文字，平台可能不会认为用户真的输入了内容。所以我们要做的是：等待页面加载，找到正确的输入区域，用接近真实用户操作的方式填入标题、正文和图片。这个阶段也加入了 AI 写作助手、平台预览、图片上传、图片拖拽缩放和多轮界面优化。

第三个阶段是 2026 年 5 月 31 日，项目从功能堆叠进入产品闭环。这个阶段我们补齐了微博支持，形成了五个平台的发布链路，同时增加了一键发布、全平台发布、并行发布、取消发布、发布记录、演示模式和进阶指引。也就是说，这时项目已经不只是“我能填进某个平台”，而是变成了一个完整流程：写内容、AI 辅助、平台预览、扩展发布、结果记录。

这个阶段还有一个很重要的安全改动，就是 AI Key 从前端迁移到了后端。因为如果把 Key 放在浏览器里，别人很容易从前端代码或网络请求里看到。后端代理的做法是：前端只请求我们自己的后端，真正访问 AI 平台的动作由后端完成。这样更安全，也方便统一管理超时、错误处理和不同 AI 供应商的差异。

第四个阶段是 2026 年 6 月 8 日之后，项目进入公开整理和稳定性修复阶段。我们把仓库整理成对外可看的版本，同时继续修复演示中会影响体验的问题。比如左侧导航栏原来会跟着页面滚动，我们把它改成固定在左侧；AI 请求原来不同地方超时时间不统一，我们统一成 2 分半；不同 AI 供应商的模型参数、接口地址和流式输出也做了兼容。

接下来讲一下技术架构。整个系统可以分成四层。

第一层是 Web 前端，也就是用户直接看到的界面。它负责文章编辑、平台选择、效果预览、AI 辅助写作和发布状态展示。我们用了 React、TypeScript、Vite、Tiptap、Zustand 和 Tailwind CSS。这里面 React 负责构建页面，TypeScript 负责减少类型错误，Tiptap 负责富文本编辑，Zustand 负责状态管理，Tailwind 负责样式。

第二层是 Web 后端。后端用 Node.js、Express、Prisma、SQLite 和 Zod。它主要做几件事：保存内容数据、提供接口、校验参数、记录发布状态、代理 AI 请求。这里后端就像一个调度中心，前端不直接操作数据库，也不直接访问 AI 平台，而是通过后端统一处理。

第三层是 Chrome 扩展。它是真实发布的关键。因为前端页面不能直接控制知乎、小红书、微博这些网站，但 Chrome 扩展在用户授权后，可以通过 content script 进入目标网页。扩展里有 background 负责接收任务，content script 负责在目标平台页面里执行填充和点击。

第四层是目标平台，也就是微信公众号、知乎、B 站、小红书、微博。每个平台都有自己的页面结构和编辑器逻辑，所以我们需要分别适配。

整个发布流程可以这样理解：用户在 Web 页面写好文章，点击发布；系统生成一个发布任务；Chrome 扩展拿到任务后进入目标平台；content script 等待页面加载完成；找到标题、正文、图片上传和发布按钮；模拟用户输入；最后把成功或失败结果传回来。

这个项目有三个核心技术原理。

第一个是统一内容模型和平台适配器。统一内容模型就是把文章先抽象成标题、正文、图片、标签、发布状态这些通用字段。平台适配器就是针对每个平台写不同的处理方式。这样做的好处是，系统主流程保持稳定，平台差异被放到适配器里处理。

第二个是 Chrome 扩展真实发布。这个原理的重点是权限。Web 前端不能直接操作其他网站，但是扩展可以在用户授权后进入目标网站页面。所以我们用扩展来执行真实平台上的填充和点击动作。扩展不是绕开平台，而是在用户已经登录的平台页面里自动完成用户本来要手动做的事情。

第三个是 AI 后端代理和供应商适配。不同 AI 平台接口并不完全一样，有的平台参数叫 `max_tokens`，有的平台叫 `max_completion_tokens`；有的平台对温度参数有限制；有的平台流式输出格式也不一样。如果这些差异都放在前端，前端会非常混乱，也不安全。所以我们把它们统一收敛到后端的 Provider Adapter 里。前端只调用统一接口，后端负责翻译成不同供应商能识别的请求。

这个项目最大的难点不是写一个漂亮页面，而是处理真实平台的不确定性。真实平台页面会更新，编辑器内部状态不容易控制，图片上传是异步的，登录状态也可能变化。所以我们的技术路线是：主流程统一，平台细节适配；能自动化的自动化，不能稳定自动化的地方保留提示和演示兜底。

目前 MultiPublish 已经形成了“内容编辑、AI 辅助、平台预览、扩展发布、状态记录”的闭环。后续我们会继续做三类事情：第一是稳定性，比如登录检测、失败重试和端到端测试；第二是产品能力，比如定时发布、团队协作、素材库和数据统计；第三是架构升级，比如把平台适配器插件化，让未来接入新平台更快。

最后用一句话总结：MultiPublish 的价值不是简单地把内容复制到多个平台，而是把多平台发布背后的重复流程、平台差异和 AI 辅助能力统一到一个可扩展的系统里。

## 10. 可能被问到的问题

### Q1：为什么不用各个平台官方 API？

很多内容平台的发布 API 不公开，或者权限申请复杂，而且不同平台开放程度不一样。Chrome 扩展方案可以复用用户自己的登录状态，更接近真实发布流程，也更适合做原型验证。

### Q2：自动发布会不会不稳定？

会有这个风险，因为平台页面会更新。所以我们的路线不是把选择器写死后就结束，而是要继续做页面变化检测、失败原因识别、重试机制和平台适配器配置化。

### Q3：AI 在项目里起什么作用？

AI 不是替代发布流程，而是提升内容生产效率。它可以辅助生成标题、摘要、正文改写、平台风格适配和标签建议。技术上通过后端代理接入，避免前端暴露密钥。

### Q4：这个项目和普通 CMS 有什么区别？

普通 CMS 更多是管理内容，MultiPublish 的重点是跨平台分发。它不仅管理文章，还要处理不同平台编辑器、登录态、图片上传和发布状态。

### Q5：后续最重要的技术工作是什么？

最重要的是稳定性和可维护性。具体包括平台适配器插件化、发布任务队列、失败重试、端到端测试、登录状态检测和更细的错误诊断。

### Q6：为什么还要做后端，前端不能直接调 AI 吗？

技术上前端可以直接调，但不建议这么做。因为 API Key 放在前端容易泄露，用户打开浏览器开发者工具就可能看到请求信息。后端代理可以保护密钥，也可以统一处理超时、错误、模型参数和费用控制。

### Q7：为什么不用一个通用脚本搞定所有平台？

因为各个平台的页面结构差异很大。一个平台可能是普通输入框，另一个平台可能是富文本编辑器，还有的平台由 React 状态控制。如果用一个通用脚本，稳定性会很差。所以我们的做法是“统一主流程 + 每个平台单独适配”。

### Q8：这个项目最有技术含量的地方在哪里？

不是单个页面，而是跨环境协作。Web 前端、后端、浏览器扩展、真实平台页面、AI 服务都在不同环境里运行，项目要把这些环境串成一个稳定流程，这就是主要技术难点。

## 11. 名词注释：小白版

这一部分是给你备讲用的。正式演讲时不需要全部念，但被问到时可以用这里的解释。

### 前端

前端就是用户能看到和操作的部分，比如页面、按钮、编辑器、预览界面。MultiPublish 的前端就是那个内容管理控制台。

### 后端

后端是用户看不见但负责处理数据和业务逻辑的部分。比如保存文章、记录发布状态、转发 AI 请求，都属于后端工作。

### API

API 可以理解成“程序之间说话的接口”。前端想让后端保存文章，就调用后端提供的 API；后端想请求 AI 服务，也要调用 AI 平台的 API。

### 接口

接口就是约定好的入口。比如 `/api/ai/chat` 这个接口表示“我要调用 AI 聊天能力”。调用方按照约定传数据，服务方按照约定返回结果。

### React

React 是一个前端框架，用来构建复杂页面。它可以把页面拆成很多组件，比如侧边栏、编辑器、按钮、卡片等。

### TypeScript

TypeScript 是 JavaScript 的增强版，多了类型检查。它能提前发现一些变量类型、函数参数不匹配的问题，让项目更不容易出低级错误。

### Vite

Vite 是前端构建工具。它负责启动开发服务、打包代码、让 React 项目更快运行。

### Zustand

Zustand 是前端状态管理工具。状态可以理解成“当前页面记住的数据”，比如当前选中的平台、文章内容、发布进度。

### Tiptap

Tiptap 是富文本编辑器框架。它负责让用户不只是输入纯文本，还能处理标题、段落、图片、加粗、列表等格式。

### ProseMirror

ProseMirror 是 Tiptap 底层依赖的编辑器引擎。可以理解成 Tiptap 背后真正管理富文本结构的核心。

### Tailwind CSS

Tailwind 是样式工具，用来快速写页面样式，比如颜色、间距、布局、字体大小。

### Node.js

Node.js 让 JavaScript 可以运行在服务器端。MultiPublish 的后端就是用 Node.js 生态构建的。

### Express

Express 是 Node.js 里的后端框架，用来写 API。比如保存内容、获取文章列表、调用 AI，都可以通过 Express 定义接口。

### Prisma

Prisma 是数据库工具，也叫 ORM。它让代码可以用更安全、更结构化的方式操作数据库，而不是到处手写 SQL。

### SQLite

SQLite 是一个轻量级数据库。它适合本地项目、演示项目和中小规模应用，不需要单独安装复杂数据库服务。

### Zod

Zod 是参数校验工具。比如接口要求标题必须是字符串、平台必须是指定值，Zod 可以在请求进入业务逻辑前先检查。

### Chrome Extension

Chrome Extension 就是 Chrome 浏览器扩展。MultiPublish 用扩展来进入真实平台页面，完成自动填充和发布。

### Manifest V3 / MV3

MV3 是 Chrome 扩展的新规范。它规定扩展怎么声明权限、怎么运行后台脚本、怎么和网页交互。

### Plasmo

Plasmo 是开发 Chrome 扩展的框架。它让扩展项目更像现代前端项目，开发体验更好。

### background

background 是扩展的后台脚本，可以理解成扩展的大脑。它负责接收任务、管理扩展状态、协调不同页面。

### content script

content script 是注入到目标网页里的脚本。比如打开知乎编辑页面后，content script 可以在这个页面里找到标题输入框和正文编辑器，然后填入内容。

### chrome.storage.local

这是 Chrome 扩展提供的本地存储。MultiPublish 用它在 Web 端、扩展后台和 content script 之间传递任务和结果。

### DOM

DOM 是浏览器对网页结构的表示。网页里的按钮、输入框、图片、标题，在程序里都可以看成 DOM 节点。

### 选择器

选择器用来定位网页元素。比如要找到“发布按钮”，程序需要用选择器告诉浏览器目标按钮在哪里。

### MutationObserver

MutationObserver 是浏览器提供的监听机制，可以观察网页结构变化。目标平台页面经常异步加载，MutationObserver 可以帮助我们等编辑器出现后再执行填充。

### Shadow DOM

Shadow DOM 是一种把网页组件内部结构隔离起来的技术。它会让普通选择器不容易找到里面的元素，所以自动化发布时会更麻烦。

### React Fiber

React Fiber 是 React 内部管理页面状态和渲染的机制。有些平台页面由 React 控制，单纯改 DOM 不一定生效，需要让 React 的状态也感知到变化。

### 富文本

富文本就是带格式的内容，比如标题、加粗、图片、列表。和纯文本相比，富文本更接近公众号或文章编辑器里的内容。

### 适配器

适配器就是“转换层”。系统内部用统一格式处理内容，到了某个平台时，适配器负责把统一内容转换成这个平台需要的格式和操作方式。

### Provider Adapter

Provider Adapter 是 AI 供应商适配器。不同 AI 平台接口不一样，这个适配器负责把系统统一请求转换成各个平台能理解的请求。

### SSE / 流式输出

SSE 是一种服务器持续向前端推送内容的方式。AI 打字机一样一个字一个字返回结果，通常就会用流式输出。

### API Key

API Key 是调用第三方服务的密钥，类似通行证。它不能直接放在前端，否则容易泄露。

### 代理

代理就是中间转发层。前端不直接访问 AI 平台，而是先访问自己的后端，再由后端访问 AI 平台，这就是后端代理。

### 超时时间

超时时间是请求最多等待多久。比如设置成 2 分半，表示 AI 请求超过 2 分半还没返回，就认为失败，避免一直卡住。

### 并行发布

并行发布就是多个平台同时发，而不是一个发完再发下一个。好处是速度快，难点是状态管理和错误处理更复杂。

### 任务 ID

任务 ID 是每次发布任务的唯一编号。它用来区分不同文章、不同平台、不同发布状态，避免结果混在一起。

### 端到端测试

端到端测试就是从用户操作开始，把完整流程跑一遍。比如创建文章、点击发布、扩展填充平台、返回结果，这整条链路都测。

### CSP

CSP 是内容安全策略，用来限制页面可以连接哪些外部地址。它能减少一部分安全风险，但配置不对也会导致接口请求被拦截。

### CORS

CORS 是跨域资源共享。当前端和后端地址不一样时，浏览器会检查后端是否允许这个前端访问。

### CRUD

CRUD 是创建、读取、更新、删除的缩写。内容管理里的新建文章、查看文章、修改文章、删除文章就是 CRUD。

### SaaS

SaaS 是软件即服务。简单说，就是把软件部署到云端，用户通过账号在线使用，而不是只在本地运行。

## 12. 一页式总结

**项目目标**：一次编辑，多平台适配，统一发布。
**核心架构**：Web 前端 + Node 后端 + Chrome 扩展 + 平台适配器 + AI 后端代理。
**项目历程**：基础架构 -> 真实发布 -> 产品闭环 -> 稳定性修复。
**技术重点**：富文本编辑、扩展注入、平台适配、任务状态同步、AI 多供应商兼容。
**最大难点**：真实平台页面差异和不确定性。
**后续路线**：稳定性、协作能力、数据复盘、插件化、多平台扩展。
