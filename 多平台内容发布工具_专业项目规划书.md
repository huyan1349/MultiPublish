# 多平台内容发布工具：72小时实战项目规划书

> 面向公众号、知乎、B站、小红书的内容适配、预览与模拟发布系统

**版本：** V2.0 专业实操版

---

## 一、项目摘要

本项目拟实现一款面向创作者与新媒体运营者的多平台内容发布工具。用户在系统中输入一份原始内容，系统将内容解析为统一结构化模型，并根据公众号、知乎、B站、小红书等平台的内容形态、表达风格和字段约束，自动生成不同平台的标题、正文、摘要、标签、封面文案和视频简介。系统支持平台预览、人工微调、格式校验、模拟一键发布、发布记录追踪，并通过平台适配器架构预留更多平台扩展能力。

项目不追求在72小时内接入所有真实平台 API，而是优先完成“输入 - 适配 - 预览 - 校验 - 模拟发布 - 记录”的闭环。真实发布能力通过 Publisher 接口预留，后续可替换为官方 API、开放平台接口、企业授权接口或受控浏览器自动化方案。

项目关键词：一次创作、多端适配、内容结构化、平台适配器、模拟发布、持续 PR、可复现 Demo。

## 二、议题要求拆解与交付物对齐

题目核心要求并不只是“做一个编辑器”，而是要体现三个能力：
1. 用户输入内容：必须有清晰的内容输入界面，建议支持标题、正文、标签、封面、内容类型选择。
2. 自动适配平台：必须能针对不同平台生成不同风格与格式的版本，不能只是复制同一段文本。
3. 一键发布或模拟发布：必须有发布动作、状态反馈和发布记录，即使是模拟发布也要表现得像真实业务流程。
4. 扩展更多平台架构：必须在代码和文档中体现插件化、适配器化、配置化，而不是把所有平台逻辑硬编码在一个函数里。

比赛提交物对齐：
- 公开 GitHub/Gitee 仓库：仓库需在开题后创建，提交截止后可公开访问。
- Demo 视频：需带声音讲解，覆盖主要功能和效果，链接放在 README 显眼位置。
- README 文档：必须说明项目背景、功能、技术栈、启动方式、依赖、原创部分、PR 记录、Demo 链接。
- 开发过程：必须全周期持续交付，不允许最后一天一次性导入所有代码。

## 三、市场需求分析

多平台内容分发已经成为内容创作者、品牌运营、校园团队、知识博主和产品宣传人员的高频工作。中国内容生态呈现明显的平台分化：公众号偏长文与私域沉淀，知乎偏问答与理性讨论，B站偏视频与知识型社区，小红书偏生活方式、搜索种草与图文/短视频笔记。创作者如果逐个平台手动改标题、调语气、补标签、换封面、复制排版，会产生大量重复劳动。

需求痛点包括：
1. 平台格式差异大：同一内容在不同平台需要不同标题结构、段落长度、标签数量、封面比例和表达方式。
2. 内容风格差异大：公众号要求正式完整，知乎要求逻辑清晰，小红书要求轻快种草，B站要求视频化表达。
3. 手工发布成本高：复制粘贴、图片迁移、标签补全、排版修复和多账号切换都会消耗时间。
4. 真实发布受平台限制：部分平台没有稳定公开发文 API，真实发布需要账号资质、授权、审核和风控处理。
5. 现有工具同质化明显：不少工具强调“一键分发”，但对“平台风格重写、结构化适配、可扩展架构”的展示不足。

本项目的差异化机会：在72小时竞赛中，不与成熟商业工具硬拼全平台数量，而是突出“自动适配质量 + 工程架构清晰 + 可演示闭环 + 持续交付规范”。

## 四、多平台调研：平台定位与适配策略

| 平台 | 内容定位 | 适配目标 | MVP 输出字段 | 注意事项 |
|---|---|---|---|---|
| 微信公众号 | 长文、品牌沉淀、私域传播 | 正式、完整、层次清楚 | 标题、摘要、正文HTML/Markdown、封面、作者 | 真实发布涉及公众号资质、素材上传、草稿和发布流程，MVP建议模拟发布 |
| 知乎 | 问答、观点、知识解释 | 结论先行、逻辑链完整 | 问题式标题、回答式正文、话题标签 | 公开稳定发文接口有限，避免依赖非官方接口作为核心能力 |
| B站 | 视频、知识分享、项目展示 | 视频标题、简介、分段时间轴、标签 | 视频标题、简介、分区、标签、时间轴 | MVP可模拟视频投稿，不强制真实上传视频 |
| 小红书 | 生活方式、种草、经验分享、搜索型内容 | 短标题、口语化、emoji、标签、封面文案 | 标题、笔记正文、话题标签、封面文案 | 真实发布接口门槛较高，MVP重点展示风格适配与模拟发布 |

平台调研结论：
1. 四个平台的核心差异不是“字段不同”，而是“内容消费语境不同”。
2. 系统应先把内容拆成标准结构，再由平台适配器重组，而不是直接字符串替换。
3. 真实发布能力应当作为可替换模块设计，比赛阶段优先使用模拟发布，降低账号、API、风控和审核风险。
4. 平台扩展时应新增 Adapter 与配置，不应修改核心业务流程。

## 五、竞品与工具分析

| 工具/类型 | 代表 | 优势 | 不足 | 对本项目启发 |
|---|---|---|---|---|
| 海外社媒管理工具 | Buffer、Hootsuite、Metricool、Later | 排期、日历、团队协作、数据分析成熟 | 对中文内容生态和公众号/知乎/小红书支持有限 | 学习其“发布日历、队列、状态管理” |
| 国内一键分发工具 | 蚁小二、简媒、微小宝等 | 平台覆盖多，多账号管理强 | 商业工具重功能数量，竞赛中难复刻全部平台 | MVP不要拼平台数量，要拼适配质量与架构 |
| 开源同步工具 | Wechatsync 等 | 开源、浏览器本地执行、支持多平台同步 | 更偏同步/迁移，风格重写与可视化工作流有限 | 学习其“本地安全、平台插件、格式转换” |
| AI写作工具 | ChatGPT、Claude、各类新媒体助手 | 文案生成能力强 | 缺少发布工作流和平台状态管理 | 将AI作为适配模块，而不是项目全部 |

竞品分析后的产品定位：
本项目不是“又一个排期工具”，而是“平台内容适配中台”。它应该展示：同一原始内容经过规则引擎和风格转换后，能够生成明显不同的公众号版、知乎版、B站版和小红书版，并且具备可扩展的发布链路。

## 六、产品功能设计

MVP 必做功能：
1. 内容输入：标题、正文、标签、封面、内容类型。
2. 平台选择：公众号、知乎、B站、小红书多选。
3. 自动适配：生成不同平台的标题、正文、标签、摘要/简介/封面文案。
4. 平台预览：以卡片或 Tab 形式展示每个平台版本。
5. 格式校验：检查标题长度、标签数量、正文是否为空、是否需要封面。
6. 人工编辑：允许用户修改各平台生成结果。
7. 模拟发布：点击发布后生成发布任务和结果。
8. 发布记录：展示平台、状态、时间、模拟链接、失败原因。
9. README 与 Demo 支撑：仓库内能清楚说明如何运行、如何演示。

增强功能，可视时间选择：
1. AI 标题生成与多版本候选。
2. 图片/封面文案生成建议。
3. 发布日历。
4. 多账号配置。
5. 内容模板库。
6. 草稿保存与历史版本。
7. 真实 API 发布预留。

不建议72小时内强做：
1. 全平台真实登录发布。
2. 高复杂权限系统。
3. 商业级数据分析。
4. 复杂视频上传和转码。
5. 跨平台评论/私信管理。

## 七、系统架构设计

推荐采用前后端分离 + 模块化后端架构。核心架构如下：

用户界面层：内容编辑器、平台选择、适配预览、发布中心、配置页面。
业务服务层：内容管理服务、内容解析服务、平台适配服务、发布服务、记录服务。
平台适配层：WeChatAdapter、ZhihuAdapter、BilibiliAdapter、XiaohongshuAdapter。
发布执行层：MockPublisher、RealPublisher 预留、任务状态管理。
数据层：SQLite/PostgreSQL，保存原始内容、平台版本、发布记录、平台配置。

架构关键原则：
1. 标准内容模型：系统内部统一使用 StandardContent 表示内容。
2. 适配器模式：每个平台独立实现 validate、transform、preview、publish。
3. 配置化约束：标题长度、标签数量、是否需要封面等应放在 platform config。
4. 模拟发布优先：保证演示稳定，避免被真实平台权限卡死。
5. 主分支随时可运行：每个 PR 合并后都要保证 demo 不崩。

## 八、核心数据模型

标准内容模型 StandardContent：
```ts
type StandardContent = {
  id: string;
  title: string;
  summary?: string;
  rawMarkdown: string;
  blocks: ContentBlock[];
  coverImage?: string;
  tags: string[];
  contentType: 'article' | 'note' | 'video_script' | 'tutorial' | 'review';
  createdAt: string;
  updatedAt: string;
};
```

内容块模型 ContentBlock：
```ts
type ContentBlock =
  | { type: 'heading'; level: 1 | 2 | 3; text: string }
  | { type: 'paragraph'; text: string }
  | { type: 'image'; url: string; caption?: string }
  | { type: 'quote'; text: string }
  | { type: 'list'; items: string[] };
```

平台输出模型 PlatformOutput：
```ts
type PlatformOutput = {
  id: string;
  contentId: string;
  platform: 'wechat' | 'zhihu' | 'bilibili' | 'xiaohongshu';
  title: string;
  body: string;
  summary?: string;
  tags: string[];
  coverImage?: string;
  extra: Record<string, unknown>;
  validation: ValidationResult;
  status: 'draft' | 'ready' | 'publishing' | 'published' | 'failed';
};
```

发布记录 PublishRecord：
```ts
type PublishRecord = {
  id: string;
  outputId: string;
  platform: string;
  status: 'success' | 'failed';
  message: string;
  mockUrl?: string;
  publishedAt: string;
};
```

## 九、平台适配器接口设计

必须在代码中体现统一接口。建议如下：

```ts
export interface PlatformAdapter {
  platform: string;
  displayName: string;
  validate(content: StandardContent): ValidationResult;
  transform(content: StandardContent): Promise<PlatformOutput>;
  getPreview(output: PlatformOutput): PreviewData;
}
```

适配器工厂：
```ts
export class AdapterRegistry {
  private adapters = new Map<string, PlatformAdapter>();

  register(adapter: PlatformAdapter) {
    this.adapters.set(adapter.platform, adapter);
  }

  get(platform: string) {
    const adapter = this.adapters.get(platform);
    if (!adapter) throw new Error(`Unsupported platform: ${platform}`);
    return adapter;
  }

  list() {
    return Array.from(this.adapters.values());
  }
}
```

新增平台步骤：
1. 在 `platform-configs` 中新增平台约束。
2. 新建 `xxx.adapter.ts` 实现统一接口。
3. 在 `adapter.registry.ts` 中注册。
4. 新增预览组件或复用通用预览组件。
5. 新增单元测试与 README 说明。

这部分是项目工程亮点，答辩时必须强调。

## 十、平台适配规则细化

公众号适配：
- 标题：正式、完整、适合长文传播。
- 摘要：80-120字导语。
- 正文：保留完整结构，增加小标题、加粗重点和分割线。
- 标签：不强调大量标签，可保留2-4个主题词。
- 封面：生成一句稳重型封面文案。

知乎适配：
- 标题：问题式或观点式。
- 开头：先给结论，再展开背景与分析。
- 正文：采用“结论 - 背景 - 问题 - 方案 - 总结”的结构。
- 标签：3-5个知识型话题。

B站适配：
- 标题：视频化、看点明确。
- 简介：不照搬文章，改成视频简介。
- 时间轴：根据章节自动生成 00:00、01:00、02:30 等模拟时间点。
- 标签：5-8个视频标签。
- 额外字段：分区、是否原创、封面文案。

小红书适配：
- 标题：短、直接、强调痛点或收益。
- 正文：口语化、短段落、适量 emoji、分点表达。
- 标签：5-10个话题标签。
- 封面文案：一句强吸引力短句。

注意：所有规则应为“建议约束”，避免写死真实平台可能变化的精确规则。比赛项目应体现可配置能力，而不是依赖不可控平台规则。

## 十一、技术栈分析与推荐

推荐技术栈：
前端：React + Vite + TypeScript + Tailwind CSS + shadcn/ui 或 Ant Design。
后端：Node.js + NestJS/Express + TypeScript，或 Python FastAPI。
数据库：SQLite 优先，PostgreSQL 作为扩展方案。
ORM：Prisma 或 Drizzle。
编辑器：Monaco Editor / Milkdown / MdEditor。
状态管理：Zustand / Redux Toolkit。
接口文档：OpenAPI / Swagger。
测试：Vitest + Playwright。
部署：Docker Compose 可选；72小时内本地启动优先。

建议选择：
- 前端：React + Vite + TypeScript + Tailwind。
- 后端：NestJS + Prisma + SQLite。
- 理由：TypeScript 前后端统一类型，适合快速开发；SQLite 降低部署复杂度；NestJS 结构清晰，方便展示架构。

不建议：
- 一上来做微服务。
- 一上来接复杂真实登录。
- 过度依赖爬虫或非官方 API。
- 只写前端假页面，没有后端数据流。

## 十三、后端 API 规划

核心接口：
```http
POST /api/contents
GET /api/contents
GET /api/contents/:id
POST /api/contents/:id/adapt
PUT /api/platform-outputs/:id
POST /api/platform-outputs/:id/publish
POST /api/publish/batch
GET /api/publish-records
GET /api/platforms
```

接口设计重点：
1. 创建内容与平台适配分离，便于保存原稿和重复生成。
2. 平台输出可编辑，体现人机协同。
3. 发布动作单独建模，便于记录状态。
4. 平台列表来自配置，不要写死在前端。
5. 批量发布可以复用单平台发布逻辑。

模拟发布返回示例：
```json
{
  "platform": "xiaohongshu",
  "status": "success",
  "mockUrl": "https://mock.contentbridge.local/xiaohongshu/post_10001",
  "message": "模拟发布成功",
  "publishedAt": "2026-05-29T12:00:00Z"
}
```

## 十四、72小时任务规划

第 1 天目标：仓库创建、基础架构和最小闭环起步。
- 创建开题后新仓库。
- 初始化 README、PR 模板、目录结构。
- 搭建前端页面骨架和后端服务。
- 完成内容输入表单。
- 完成 StandardContent 数据模型。
- 完成第一个平台适配器，例如公众号。
- 合并多个小 PR，确保主分支可运行。

第 2 天目标：核心功能成型。
- 完成知乎、B站、小红书适配器。
- 完成平台预览页面。
- 完成格式校验逻辑。
- 完成平台输出保存与编辑。
- 完成模拟发布接口。
- 完成发布记录页面。
- 增加基础测试。
- 持续拆分 PR，每个 PR 只做一件事。

第 3 天目标：打磨、演示和提交。
- 修复 Bug 和交互细节。
- 补充 README、架构图、接口说明、依赖说明。
- 录制 Demo 视频并上传外部平台。
- 在 README 显眼位置放 Demo 链接。
- 整理 PR 描述，确认 commit 时间戳均在周期内。
- 检查主分支可从零启动。
- 最终公开仓库并提交。

## 十八、测试与质量保障

必须测试：
1. 空标题、空正文不能生成发布结果。
2. 四个平台都能生成不同版本。
3. 平台适配结果可以被编辑和保存。
4. 模拟发布成功后发布记录生成。
5. 失败场景能给出错误提示。
6. 主分支从 README 命令能正常启动。

建议测试分类：
- 单元测试：适配器 transform、validate、标签生成、标题裁剪。
- 接口测试：创建内容、生成适配、模拟发布、查询记录。
- 端到端测试：从输入内容到发布记录完整跑通。

质量红线：
- 不允许主分支无法启动。
- 不允许 README 命令与实际代码不一致。
- 不允许 PR 描述空白。
- 不允许依赖未列明。
- 不允许把真实账号密码写入仓库。

## 十九、风险与注意事项

技术风险：
1. 平台真实 API 不稳定或无权限：采用 MockPublisher，真实接口只做预留。
2. AI 生成不可控：采用规则模板 + 人工编辑，不让 AI 直接发布。
3. 时间不足：优先保证闭环，不追求全平台真实发布。
4. 前后端联调耗时：尽早定义接口，使用 mock 数据并行开发。

合规风险：
1. 不采集用户真实账号密码。
2. 不绕过平台风控或验证码。
3. 不使用爬虫批量发布真实内容。
4. 不公开涉及他人知识产权的代码或素材。
5. 若使用第三方库必须在 README 中列明。

比赛风险：
1. 最后一天一次性提交会被判无效，必须持续 PR。
2. PR 描述与实际代码不符会扣分甚至无效。
3. Demo 链接隐藏太深，评委可能看不到。
4. commit 时间戳不在周期内可能无效。
5. 主分支不可运行会严重影响评分。

## 二十、评分导向下的取舍策略

作品完整度与创新性 40%：
- 必须展示完整闭环。
- 四个平台适配结果要明显不同。
- 预览页面要直观，最好能一眼看到风格差异。
- 强调内容结构化和平台适配器，不做普通表单工具。

开发过程与质量 40%：
- PR 数量与质量要充分。
- 架构要分层清晰。
- 代码要有类型、校验和错误处理。
- README 能让评委复现。
- 至少有部分测试。

演示与表达 20%：
- Demo 视频要有声音。
- 演示路径要顺滑，不要现场调试。
- 讲清楚“为什么模拟发布也是有效实现”。
- 讲清楚“如何扩展新平台”。

最终策略：
不要做一个“大而空”的平台，要做一个“小而完整、架构漂亮、演示稳定”的产品。

## 二十一、最终开发路线图

推荐最终交付版本包含：
1. 一个可运行前端页面。
