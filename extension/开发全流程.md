## 1. 环境准备

```bash
# 必须使用 pnpm（Plasmo 强依赖）
npm install -g pnpm

# Node.js >= 18
node -v
```

## 2. 项目初始化

```bash
# 在当前仓库中创建 extension 目录
mkdir extension && cd extension

# 手动创建 package.json、tsconfig.json
# 核心依赖：
#   plasmo — Chrome 扩展开发框架
#   react + react-dom — Sidepanel UI
#   @plasmohq/storage — chrome.storage 封装
#   zustand — 状态管理
#   lucide-react — 图标

pnpm install
pnpm approve-builds sharp esbuild @swc/core @parcel/watcher lmdb msgpackr-extract
```

**关键依赖清单（package.json）：**

| 依赖 | 用途 |
|------|------|

| `@plasmohq/storage` | chrome.storage.local 封装 API |
| `react` / `react-dom` | Sidepanel UI |
| `zustand` | 编辑器草稿 + 内容/记录管理 |
| `lucide-react` | 图标 |

## 3. 架构设计

```
extension/src/
├── background.ts              # Service Worker（消息路由 + 发布调度）
├── sidepanel.tsx              # 侧边栏入口（单文件 SPA，4 个页面状态）
├── contents/                  # Content Scripts（注入到目标页面执行）
│   ├── wechat.ts              # 微信公众号 — all_frames 穿透 UEditor iframe
│   ├── zhihu.ts               # 知乎 — ClipboardEvent 模拟 Draft.js
│   ├── bilibili.ts            # B站 — Quill.js DOM 注入
│   └── xiaohongshu.ts         # 小红书 — textarea 原生 setter 注入
├── shared/types.ts            # 共享类型定义
└── sidepanel/
    ├── adapters/              # 4 个平台适配器（从 backend 迁移，纯 TS）
    │   ├── PlatformAdapter.ts  # 适配器接口定义
    │   ├── AdapterFactory.ts   # 注册工厂
    │   ├── parserService.ts    # Markdown → ContentBlock[] 解析器
    │   ├── wechat/WechatAdapter.ts
    │   ├── zhihu/ZhihuAdapter.ts
    │   ├── bilibili/BilibiliAdapter.ts
    │   └── xiaohongshu/XiaohongshuAdapter.ts
    ├── stores/contentStore.ts  # Zustand store（chrome.storage 持久化）
    └── styles/index.css        # 暗夜编辑室主题
```

### 核心数据流

```
用戶在 Sidepanel 编辑 → 适配器 transform → Content Script 注入到平台编辑器
                          │
                          ▼
              Background SW（storage 信号）
                          │
            ┌─────────────┼─────────────┐
            ▼             ▼             ▼
        公众号CS      知乎CS       B站CS / 小红书CS
       UEditor      Draft.js     Quill.js / textarea
       iframe穿透   粘贴模拟     DOM注入 / setter
            │             │             │
            └─────────────┼─────────────┘
                          ▼
              写结果到 storage.local
                          ▼
              Background 收到 → 报告成功/失败
```

### 4.4 发布机制（经过 3 次迭代）

| 版本 | 方案 | 问题 | 状态 |
|------|------|------|------|
| v1 | Background `sendMessage` → Content Script `onMessage` | 时序问题，监听器未注册时消息已到 | ❌ |
| v2 | `chrome.scripting.executeScript` 注入大函数 | 序列化不稳定，异步返回值不可靠 | ❌ |
| **v3** | **Storage 信号机制** | — | ✅ 当前 |

### 4.5 Content Script 注入策略

| 平台 | 编辑器技术 | 注入方式 |
|------|-----------|---------|
| 公众号 | UEditor（iframe 嵌套） | `all_frames: true` + `MutationObserver` 轮询 `[contenteditable]` |
| 知乎 | Draft.js（不可变状态） | `ClipboardEvent` 粘贴模拟 + `DataTransfer` |
| B站 | Quill.js | `innerHTML` 直接写入 + 原生 setter |
| 小红书 | React 受控组件 | `getOwnPropertyDescriptor` 绕过 React 直接设 `value` |

## 7. 发布机制详解

### 7.1 完整流程

```
┌─────────────────────────────────────────────────────────┐
│ Sidepanel                                               │
│                                                         │
│  用户点击「发布到公众号」                                  │
│    │                                                    │
│    ▼                                                    │
│  chrome.runtime.sendMessage({                           │
│    type: 'PUBLISH_TO_PLATFORM',                         │
│    payload: { platform, content }                       │
│  })                                                     │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│ Background Service Worker                               │
│                                                         │
│  ① 收到消息                                              │
│  ② chrome.storage.local.set({ contentbridge_fill })     │
│  ③ chrome.tabs.create({ url: 平台编辑器URL })           │
│  ④ 监听 chrome.storage.onChanged                        │
│     等待 contentbridge_result                            │
│  ⑤ 收到结果 → chrome.tabs.update 激活标签页              │
│  ⑥ sendResponse 返回结果给 Sidepanel                    │
└────────────────────┬────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────────┐
│ Content Script（在平台页面上运行）                         │
│                                                         │
│  ① 页面加载完成（document_idle）                         │
│  ② 检查 chrome.storage.local → 发现 contentbridge_fill  │
│  ③ 移除 fill 数据（防止重复）                             │
│  ④ MutationObserver 轮询编辑器 DOM（最长 8s）            │
│  ⑤ 编辑器出现 → 填充内容                                  │
│  ⑥ chrome.storage.local.set({ contentbridge_result })   │
└─────────────────────────────────────────────────────────┘
```

### 7.2 微信特殊处理

微信编辑器是双层结构：主页 `mp.weixin.qq.com` → 内嵌 UEditor iframe。

```typescript
// Plasmo config
export const config: PlasmoCSConfig = {
  matches: ['https://mp.weixin.qq.com/*'],
  all_frames: true,  // 注入到主页 + iframe
  run_at: 'document_idle',
};

// Content Script 逻辑
if (window.self !== window.top) {
  // 在 iframe 里 → 直接填充 contenteditable
  const ed = document.querySelector('[contenteditable="true"]');
  ed.innerHTML = body;
} else {
  // 在主页 → 填标题 + 尝试访问 iframe.contentDocument
  document.querySelector('#title').value = title;
}
```

### 7.3 为什么不用 executeScript

| 方案 | 可靠性 | 原因 |
|------|--------|------|
| `executeScript` | ❌ | 大函数序列化 → `toString()` 不稳定；异步返回值处理复杂；allFrames 返回值合并逻辑不明确 |
| **Storage 信号** | ✅ | Content Script 主动拉取 → 无时序问题；MutationObserver 保证 DOM 就绪；结果写入 storage → Background 被动监听 |

## 9. 调试指南

### 9.1 查看 Sidepanel 日志
1. 打开 Sidepanel
2. 右键 → 检查 → Console

### 9.2 查看 Background Service Worker 日志
1. `chrome://extensions/`
2. 找到 ContentBridge → 点击 "Service Worker" 链接
3. 在打开的 DevTools 中查看 Console

### 9.3 查看 Content Script 日志
1. 打开目标平台页面（如 `mp.weixin.qq.com`）
2. F12 → Console
3. 筛选框输入 `contentbridge` 查看相关日志

### 9.4 查看 Storage 数据
1. F12 → Application → Storage → chrome.storage.local
2. 可以看到 `contentbridge_fill`（待处理）和 `contentbridge_result`（已处理）

### 9.5 重新加载扩展
- 修改代码后：`pnpm build`（或 `pnpm dev` 自动热更新）
- `chrome://extensions/` → 点 🔄 刷新按钮
- **重要**：Sidepanel 和 Content Script 需要重新打开/刷新页面才能加载

## 10. 常见问题

### Q: 填充后编辑器没反应？
A: 平台可能升级了编辑器 DOM 结构。F12 检查实际元素，更新 Content Script 中的 CSS 选择器。

### Q: 公众号提示「未找到编辑器」？
A: 确认已登录 `mp.weixin.qq.com`，且打开的是图文编辑页面（不是首页）。检查 UEditor iframe 是否已加载。

### Q: 知乎粘贴后内容消失？
A: Draft.js 不可变状态模型的特性。确保使用 ClipboardEvent 模拟而非 innerHTML 直接写入。

### Q: 小红书填不进去？
A: 小红书使用 React 受控组件。必须用 `Object.getOwnPropertyDescriptor(HTMLTextAreaElement.prototype, 'value').set.call(el, value)` 绕过，再 dispatch `input` 事件。

### Q: 扩展如何升级？
A: `pnpm build` → `chrome://extensions/` 刷新 → 重新打开 Sidepanel。
